Step 0: Site Structure (Preliminary Step)

Before building the site with prompts, you must create a precise structure. I have prepared a custom model (Gem) that will do the heavy lifting and set up the foundation. Extract the structure questionnaire from there and return here.

1. Creating the Sections and Components List

A prompt designed to define the structural layout of the site from the completed structure questionnaire.

להלן שאלון אפיון לדף הבית: [צירוף שאלון אפיון מהGem] בהתבסס על הנתונים שאספק לך, בנה עבורי אפיון מבני מלא לדף הבית של האתר. המטרה שלך היא להגדיר בצורה מדויקת ומקצועית: - רשימה מסודרת של כל הסקשנים (Sections) שצריכים להופיע בדף הבית, לפי סדר לוגי שמוביל את המשתמש לפעולה. - עבור כל סקשן – רשימה מלאה ומפורטת של כל הרכיבים (Components) שחייבים להופיע בו. פירוט מלא של כל רכיב: 1. סוג הרכיב (כותרת, טקסט, כפתור, תמונה, אייקון, טופס, כרטיס, רשימה, וידאו, המלצה וכו') 2. מטרת הרכיב (מה הוא אמור לגרום למשתמש להבין / להרגיש / לעשות) 3. האם הוא רכיב מניע לפעולה (CTA) ואם כן – איזה סוג פעולה 4. היררכיה טקסטואלית (H1 / H2 / H3 / פסקה / תיאור קצר / מיקרו-קופי) 5. אינטראקציה נדרשת (קישור, פתיחת טאב חדש, גלילה לאזור, פתיחת מודאל, שליחת טופס וכו') הנחיות חשובות: - אל תתייחס כלל לעיצוב, צבעים, פונטים או סגנון ויזואלי. - אל תציע אנימציות או אפקטים גרפיים. - התמקד אך ורק במבנה, בהיררכיה ובפסיכולוגיית המרה. - ודא שבכל אזור יש היגיון שיווקי ברור. - ודא שיש מספר נקודות CTA לאורך הדף ולא רק אחת. - חשוב במונחים של מסע משתמש (User Journey) מדף נחיתה ועד פעולה סופית. המטרה הסופית: לקבל אפיון טכני-מבני מלא שאפשר להעביר ישירות למעצב / למפתח / לכלי AI ליצירת אתר – בלי להשאיר מקום לניחושים.
Below is a homepage characterization questionnaire: [Attach structure questionnaire from Gem] Based on the information I provide, create a complete structural specification for the homepage. Your task is to define, in a precise and professional manner: - A logically ordered list of all the sections that should appear on the homepage, structured according to a clear user journey that leads toward conversion. - For each section – provide a complete and detailed list of all required components. For each component – specify: 1. Component type (headline, paragraph, button, image, icon, form, card, list, testimonial, video, etc.) 2. Purpose of the component (what the user should understand / feel / do) 3. Whether it is a call-to-action (CTA), and if so, what type of action it triggers 4. Text hierarchy (H1 / H2 / H3 / paragraph / short description / microcopy) 5. Required interaction (internal link, external link, scroll to section, modal popup, form submission, etc.) Important instructions: - Do not focus on design, colors, fonts, or visual style. - Do not suggest animations or graphic effects. - Focus strictly on structure, hierarchy, conversion psychology, and user flow. - Ensure each section has a clear strategic purpose. - Include multiple CTA points throughout the page. - Think in terms of user journey, from first impression to final action. Final goal: Deliver a complete structural blueprint that can be directly handed to a designer, developer, or AI website builder — with zero ambiguity and no missing components.

2. Generating the Image (Hero Section)

A prompt to create a concept image for the top design of the website using models like Nano Banana Pro.

להלן רשימת הרכיבים המדויקת של ה-Hero Section: [הדבק כאן את רשימת הרכיבים] צור פרומפט עבור Nano Banana Pro ליצירת תמונה של Hero Section בפורמט תצוגת דסקטופ. הנחיות מחייבות לפרומפט: - התמונה חייבת להיות בפורמט רוחבי של דסקטופ (יחס מסך רחב, לא מובייל ולא טאבלט). - יש להציג Hero Section כפריסה מלאה לרוחב (full-width desktop layout). - אין ליצור גרסת מובייל. - אין להציג מחשב, מסך, מובייל, מסגרת דפדפן, UI של מערכת ההפעלה או mockup. - אין להוסיף שורת כתובת, טאבים או אלמנטים מערכתיים. - התמונה צריכה להיות עיצוב נקי של ההירו בלבד כקנבס מלא. מבחינת רכיבים: - יש לכלול אך ורק את הרכיבים שסופקו ברשימה. - אין להוסיף רכיבים נוספים. - אין להשמיט רכיבים. - כל הטקסטים חייבים להופיע במלואם (ללא lorem ipsum או placeholder). מבחינת עיצוב: - אין להגדיר פונטים, צבעים, טיפוגרפיה או סגנון עיצובי. - אין להגדיר מיקומים מדויקים של רכיבים. - אין להכתיב פריסה. - המודל יבחר באופן עצמאי את העיצוב, המיקום וההיררכיה בצורה מקצועית. המטרה: לקבל תמונת Hero Section מלאה בפורמט דסקטופ רחב, שמכילה בדיוק את רשימת הרכיבים שסופקה, כאשר כל החלטות העיצוב נשארות בידי המודל.
Below is the exact list of components for the Hero Section: [Paste the components list here] Create a prompt for Nano Banana Pro to generate a Hero Section image in desktop display format. Mandatory constraints: - The image must be in a wide desktop aspect ratio (landscape orientation). - It must represent a full-width desktop hero layout. - Do not generate a mobile or tablet version. - Do not display any device mockup, monitor, browser frame, operating system UI, or software interface. - Do not include browser bars, tabs, or system controls. - The image must be a clean full-canvas hero section only. Component requirements: - Include only the provided list of components. - Do not add extra elements. - Do not remove any components. - All text must appear fully (no placeholder or lorem ipsum text). Design control: - Do not define fonts, colors, typography, or visual style. - Do not define positioning or layout structure. - Allow the model to determine layout, hierarchy, and visual execution autonomously. Objective: Generate a full-width desktop Hero Section image that includes exactly the specified components, with all design decisions handled by the model.

3. Building the Hero Design in Code

Once you have the design image, upload it to the AI tool and use this prompt.

No text input is needed for this step. Simply attach the generated image and copy the prompt.
צירפתי תמונה של Hero Section. בנה את ה-Hero Section בקוד אחד לאחד בהתאם לתמונה. הנחיות מחייבות: - יש לשחזר את המבנה, הפריסה וההיררכיה בדיוק כפי שמופיעים בתמונה. - אין לשנות מיקומים של רכיבים. - אין לשנות יחס גדלים בין רכיבים. - אין להוסיף רכיבים חדשים. - אין להסיר רכיבים קיימים. - אין לשנות טקסטים. - אין לנסח מחדש כותרות. - אין לשפר או “לעדכן” את העיצוב. - אין להוסיף אנימציות אם אינן מופיעות בתמונה. - אין לבצע פרשנות יצירתית. דרישות טכניות: - הפלט חייב להיות קוד מלא של ה-Hero Section בלבד. - יש להשתמש במבנה סמנטי תקין (section, h1, h2, p, button וכו'). - יש להקפיד על היררכיית HTML נכונה. - יש לכתוב קוד נקי, מסודר ומוכן לשימוש. - יש להתאים לפריסת דסקטופ רחבה. - אין לכלול מסגרת דפדפן, mockup או אלמנטים חיצוניים. המטרה: לקבל מימוש קוד מדויק ככל האפשר של ה-Hero Section כפי שהוא מופיע בתמונה, ללא כל סטייה יצירתית.
I have attached an image of a Hero Section. Build the Hero Section in code exactly as shown in the image. Strict requirements: - Replicate the structure, layout, and hierarchy exactly as displayed. - Do not change element positioning. - Do not modify size proportions. - Do not add new components. - Do not remove existing components. - Do not change any text. - Do not rephrase headlines. - Do not “improve” or redesign anything. - Do not add animations unless they clearly appear in the image. - No creative reinterpretation. Technical requirements: - Output must include only the Hero Section code. - Use proper semantic HTML structure (section, h1, h2, p, button, etc.). - Maintain correct HTML hierarchy. - Produce clean, production-ready code. - Desktop layout only (wide format). - Do not include browser frames, mockups, or external UI elements. Objective: Generate an accurate 1:1 coded implementation of the provided Hero Section with zero creative deviation.

4. Building the Rest of the Website's Components

After building the Hero in code, insert the remaining components list to complete the page in the same design language.

הנה רשימת הרכיבים של שאר הקטעים באתר (ללא ה־Hero Section): [הדבק כאן את רשימת הרכיבים] כבר בנית Hero Section בקוד. עכשיו בנה את שאר הקטעים של דף הבית בהתאם לרשימת הרכיבים, ובאותו סגנון עיצובי, שפה ויזואלית ורמת גימור של ה־Hero Section שנבנה. כללי עבודה מחייבים: אחידות מלאה עם ההירו - שמור על אותה שפת עיצוב, אותה היררכיית רכיבים, אותו “פיל” ואותה רמת פרימיום. - אל תחליף סגנון באמצע הדף. רכיבים ותוכן - בכל סקשן יש לכלול *רק* את הרכיבים שברשימה (לא להוסיף ולא להשמיט). - אם חסר טקסט מדויק לרכיב כלשהו — השתמש בטקסט הגיוני ורלוונטי שמוביל לפעולה (לא lorem ipsum), ושמור על עקביות טון מול ההירו. מבנה הדף - בנה את כל הסקשנים כ־sections מלאים, בסדר לוגי שמוביל להמרה. - דסקטופ קודם (Desktop-first) עם התאמה רספונסיבית מלאה לטאבלט ומובייל. אנימציות GSAP – חובה - שלב GSAP בצורה מקצועית, לא “קישוטית”. - חובה לכלול: 1. אנימציות Scroll-based באמצעות ScrollTrigger (כניסות, reveal, stagger, parallax עדין אם מתאים). 2. אנימציות שאינן רק לגלילה: hover אינטראקטיבי, focus states, micro-interactions לכפתורים/כרטיסים, פתיחה/סגירה של אלמנטים (למשל FAQ אם קיים), אנימציות לטעינה/מעבר. - האנימציות צריכות להיות חלקות, מדויקות, ולא להפריע לקריאות/שימושיות. ביצועים ונגישות - הקפד על ביצועים: מינימום רינדורים כבדים, שימוש חכם ב־will-change רק כשצריך, והפחתת אנימציות במובייל אם זה פוגע. - תמוך ב־prefers-reduced-motion: אם המשתמש מבקש להפחית אנימציה — צמצם/כבה אנימציות בצורה מכבדת. - שמור על סמנטיקה ונגישות: כותרות נכונות, aria כשצריך, ניווט מקלדת תקין, כפתורים אמיתיים לפעולות. תוצרים - הפלט חייב להיות קוד מלא ומוכן להרצה של שאר הסקשנים + האנימציות. - כל קטע צריך להיות מסומן בבירור עם הערה/כותרת בקוד כדי לדעת איפה מתחיל/נגמר כל section. - אל תשנה את קוד ההירו שכבר נבנה, אלא אם נדרש ממש כדי לשמור אחידות מערכתית (ואז ציין בדיוק מה שינית ולמה). מטרה: לייצר דף בית שלם (מלבד ההירו שכבר קיים) שנראה מקצועי, אחיד, “פרימיום”, עם GSAP איכותי לגלילה וגם לאינטראקציות נוספות — ומבוסס בדיוק על רשימת הרכיבים שסופקה.
Here is the component list for the rest of the website sections (excluding the Hero Section): [Paste the component list here] You already built the Hero Section in code. Now build the remaining homepage sections based on the component list, using the same design language, visual system, and level of polish as the existing Hero Section. Mandatory rules: Full consistency with the Hero - Keep the same design language, hierarchy, premium feel, and overall UI system. - Do not switch styles midway down the page. Components & content - Each section must include *only* the components in the provided list (do not add or remove items). - If any component is missing exact copy, generate realistic conversion-focused copy (no lorem ipsum) while staying consistent with the hero tone. Page structure - Build each section as a complete section in a logical conversion-driven order. - Desktop-first layout, fully responsive for tablet and mobile. GSAP animations — required - Implement GSAP professionally (not decorative only). - Must include: 1. Scroll-based animations using ScrollTrigger (reveal, stagger, subtle parallax if appropriate). 2. Non-scroll animations: interactive hover micro-interactions, focus states, button/card interactions, open/close motions (e.g., FAQ if applicable), transitions and loading/motion details. - Animations must be smooth, readable, and not harm usability. Performance & accessibility - Keep performance in mind: avoid heavy reflows, use will-change only when needed, reduce motion complexity on mobile if necessary. - Support prefers-reduced-motion: gracefully reduce/disable motion when requested. - Maintain semantic structure and accessibility: proper heading order, ARIA where needed, keyboard navigation, real buttons for actions. Deliverables - Output must be runnable, production-ready code for all remaining sections + animations. - Clearly label each section in code (comments or headings). - Do not modify the already-built hero code unless strictly necessary for system consistency — if you must, explicitly list what changed and why. Goal: Produce the rest of the homepage (excluding the existing hero) with a consistent premium UI, strong conversion flow, and high-quality GSAP animations for both scroll and non-scroll interactions — strictly based on the provided component list.

5. Exporting the Site to Lovable / Dev Tools

Once you have a fully working HTML file, this prompt will be used to import it into advanced dev tools without breaking the design.

No text input is needed for this step, simply attach the final HTML file to the AI tool along with the prompt.
צירפתי קובץ HTML של אתר קיים. המשימה שלך: לשחזר את האתר אצלך כך שייראה ויתנהג אחד לאחד כמו המקור, כולל כל הסקשנים, הרכיבים, הטקסטים, התמונות, האייקונים, הרקעים, והאנימציות — ללא שום שינוי. כללים מחייבים (אפס סטייה): דיוק 1:1 - שחזר בדיוק את המראה וההתנהגות: פריסה, מרווחים, היררכיה, סדר קטעים, גדלים, יישור, וטקסטים. - אין “שיפור”, אין “רענון”, אין “עיצוב מחדש”. לא להוסיף ולא להחסיר - אל תוסיף שום דבר שלא קיים במקור (רכיבים/סקשנים/אפקטים/טקסטים). - אל תחסיר שום דבר שקיים. - אל תשנה ניסוחים, כותרות, טקסטים, תוויות כפתורים או סדר. תמונות ונכסים - כל התמונות/אייקונים/רקעים/קבצים חיצוניים שמופיעים במקור חייבים להופיע גם אצלך. - אם נכס חסר בפועל אצלך — השתמש בפתרון זמני שמחזיק *את אותו מקום/גודל/יחס* עד שאספק את הקובץ, בלי לשנות את המבנה או העימוד. אנימציות ואינטראקציות - שחזר את כל האנימציות בדיוק כפי שבמקור: גלילה, כניסות, hover, transitions, timing, easing, stagger וכל אינטראקציה. - אל תשנה מהירות, עקומות תנועה או טריגרים. - שחזר גם אינטראקציות שאינן גלילה (hover/click/focus/open-close וכו'). רספונסיביות - אם במקור יש התנהגות רספונסיבית — שחזר אותה אחד לאחד. - אל תשנה את הדרך שבה האתר נשבר/מסתדר במסכים שונים. חופש בטכנולוגיה - אתה יכול להשתמש בכל טכנולוגיה/שיטה/ייצוא שאתה רוצה — *לא אכפת לי איך זה נבנה או איך זה מיוצא*, כל עוד התוצאה הסופית תואמת אחד לאחד למקור. בדיקת הצלחה: כשמשווים את האתר המקורי לשחזור *בצד לצד*, זה צריך להיראות ולהרגיש *זהה לחלוטין*.
I attached an HTML file of an existing website. Your task: rebuild the website so it looks and behaves exactly the same (1:1) as the original, including all sections, components, text, images, icons, backgrounds, and animations — with zero changes. Strict rules (no deviation): Exact 1:1 fidelity - Match the visuals and behavior precisely: layout, spacing, hierarchy, section order, sizing, alignment, and all copy. - No “improvements”, no refresh, no redesign. Do not add or remove anything - Do not add any new sections/components/effects/text. - Do not remove anything present in the original. - Do not rephrase copy, change labels, or reorder elements. Images & assets - All images/icons/backgrounds/external assets used in the original must appear in the rebuild. - If an asset is missing on your side, use a temporary replacement that preserves the *exact same slot/size/aspect ratio* until I provide the real file — without changing layout. Animations & interactions - Reproduce all animations exactly: scroll behavior, reveals, hover interactions, transitions, timing, easing, stagger, and triggers. - Do not change animation speed, easing curves, or triggers. - Include non-scroll interactions as well (hover/click/focus/open-close, etc.). Responsiveness - If the original has responsive behavior, recreate it exactly. - Do not change how it adapts across screen sizes. No technology constraints - You may use any technology/method/export/hosting you want — *I don’t care how it’s built*, as long as the final output matches the original 1:1. Success criteria: When comparing the original and the rebuilt version *side by side*, they should be visually and behaviorally indistinguishable.
